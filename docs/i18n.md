# Internationalization (i18n)

MiroFish-Offline ships a **bilingual German / English** UI and report pipeline.
German (`de`) is the default; English (`en`) is the fallback. The system is
designed so that adding a third language is a matter of dropping in one JSON
file and one registry entry — no code changes.

This document explains the architecture, the conventions, and how to extend it.

---

## 1. Overview

There are **two independent translation layers** that share a single source of
truth:

| Layer        | Library          | Reads from                | Used for                                   |
|--------------|------------------|---------------------------|--------------------------------------------|
| **Frontend** | `vue-i18n` (v11) | `locales/*.json`          | All UI strings in Vue components/views     |
| **Backend**  | custom `locale.py` | `locales/*.json`        | API response messages, log messages        |
| **LLM**      | `languages.json` | `locales/languages.json`  | Steering generated text into the user's language |

The key design decision: **frontend and backend read the exact same
`locales/en.json` / `locales/de.json` files.** There is one set of translation
keys for the whole application. The backend loads them with a tiny custom
resolver; the frontend loads them through `vue-i18n`.

```
locales/
├── languages.json   # registry: label + LLM instruction per language
├── de.json          # German strings   (default)
└── en.json          # English strings  (fallback)
```

Both `de.json` and `en.json` currently contain ~905 lines of keys and are kept
structurally identical (same key tree, different values).

---

## 2. The language registry — `locales/languages.json`

This small file is the canonical list of supported languages. It is read by
**all three** layers.

```json
{
  "de": {
    "label": "Deutsch",
    "llmInstruction": "Bitte antworten Sie auf Deutsch."
  },
  "en": {
    "label": "English",
    "llmInstruction": "Please respond in English."
  }
}
```

- `label` — the human-readable name shown in the language switcher dropdown.
- `llmInstruction` — a natural-language sentence injected into LLM prompts so the
  model writes generated content (reports, agent personas, bios) in the selected
  language. See [§6](#6-llm-language-steering).

**To add a language you edit this file once** (plus add the matching
`locales/<code>.json`).

## 3. Frontend (`vue-i18n`)

### Setup — `frontend/src/i18n/index.js`

```js
import { createI18n } from 'vue-i18n'
import languages from '../../../locales/languages.json'

// Eagerly glob-import every locale file EXCEPT languages.json
const localeFiles = import.meta.glob('../../../locales/!(languages).json', { eager: true })

const messages = {}
const availableLocales = []
for (const path in localeFiles) {
  const key = path.match(/\/([^/]+)\.json$/)[1]
  if (languages[key]) {                       // only register codes present in the registry
    messages[key] = localeFiles[path].default
    availableLocales.push({ key, label: languages[key].label })
  }
}

const savedLocale = localStorage.getItem('locale') || 'de'  // default = German

const i18n = createI18n({
  legacy: false,            // Composition API mode
  locale: savedLocale,
  fallbackLocale: 'en',     // missing key falls back to English
  messages
})

export { availableLocales }
export default i18n
```

Notes:
- Locale files are **glob-imported** from the shared `locales/` directory at
  `../../../locales/` (project root), not duplicated under `frontend/`. A new
  `locales/fr.json` is picked up automatically — provided `fr` is in
  `languages.json`.
- The user's choice is persisted to `localStorage` under the key `locale`.
- vue-i18n runs in **`legacy: false`** (Composition API) mode, so components use
  `useI18n()` / `$t()`.

### Registration — `frontend/src/main.js`

The plugin is registered on the app:

```js
import i18n from './i18n'
app.use(i18n)
```

### The `@` path alias — `frontend/vite.config.js`

Components import the i18n module via the `@` alias, which must be configured in
Vite:

```js
resolve: {
  alias: {
    '@': fileURLToPath(new URL('./src', import.meta.url))
  }
}
```

> ⚠️ **Gotcha:** without this alias, `import ... from '@/i18n/index.js'` fails at
> build time. This was a real bug during integration.

### Using translations in components

```vue
<script setup>
import { useI18n } from 'vue-i18n'
const { t } = useI18n()
</script>

<template>
  <h1>{{ $t('home.title') }}</h1>
  <p>{{ t('simulation.agentsCount', { count: agentNumber }) }}</p>
</template>
```

Interpolation uses the `{name}` placeholder syntax, e.g.
`"agentsCount": "{count} Agenten"`.

### The language switcher — `frontend/src/components/LanguageSwitcher.vue`

A self-contained dropdown that:
- lists every entry from `availableLocales` (derived from the registry),
- on selection sets `locale.value`, persists to `localStorage`, and updates
  `document.documentElement.lang`,
- closes on outside-click.

```js
const switchLocale = (key) => {
  locale.value = key
  localStorage.setItem('locale', key)
  document.documentElement.lang = key
  open.value = false
}
```

## 4. Backend resolver — `backend/app/utils/locale.py`

The backend does **not** use a third-party i18n library. It has a ~70-line
module that loads the same JSON files and resolves keys. It is exported from
`backend/app/utils/__init__.py`:

```python
from ..utils import t, get_locale, set_locale, get_language_instruction
```

### What it loads (at import time)

```python
_locales_dir = os.path.join(os.path.dirname(__file__), '..', '..', '..', 'locales')
# loads languages.json into _languages
# loads every other *.json into _translations[<code>]
```

So the backend reads `locales/` from the **project root** — the same directory
the frontend globs. One source of truth.

### Public API

| Function | Purpose |
|----------|---------|
| `t(key, **kwargs)` | Resolve a dotted key (`'api.ontologyNotFound'`) to a string in the current locale. Interpolates `{name}` placeholders from `**kwargs`. |
| `get_locale()` | Return the active locale code (`'de'` / `'en'`). |
| `set_locale(locale)` | Set the locale for the **current thread** (for background threads — see §5). |
| `get_language_instruction()` | Return the `llmInstruction` sentence for the active locale (for prompt injection — see §6). |

### Resolution & fallback chain (`t`)

1. Look up the key in the current locale's tree.
2. If missing, look it up in **`en`** (hard fallback).
3. If still missing, **return the key string itself** (so a missing key is
   visible, never crashes).
4. Apply `{placeholder}` substitution from `kwargs`.

### How the locale is determined (`get_locale`)

```python
def get_locale() -> str:
    if has_request_context():
        raw = request.headers.get('Accept-Language', 'de')
        return raw if raw in _translations else 'de'
    return getattr(_thread_local, 'locale', 'de')
```

- **Inside a Flask request:** read the `Accept-Language` header (validated
  against loaded translations; defaults to `de`).
- **Outside a request** (background thread): read a `threading.local` value set
  via `set_locale()`. Defaults to `de`.

The frontend sends the header automatically — see §7.

## 5. The background-thread pattern (critical)

Long-running work (graph build, simulation, report generation) runs in
**background threads** that have **no Flask request context**. The
`Accept-Language` header is therefore unavailable inside them. The pattern,
used consistently in `graph.py`, `report.py`, `simulation.py`, and
`oasis_profile_generator.py`:

```python
# --- still inside the request context ---
locale = get_locale()                 # capture the request's language NOW

def build_task():
    set_locale(locale)                # re-establish it on the worker thread
    ...                               # now t() and get_language_instruction()
                                      # resolve in the user's language
```

> ⚠️ **If you spawn a thread that calls `t()` or `get_language_instruction()`,
> you MUST `get_locale()` before the thread starts and `set_locale(...)` as the
> first line inside it.** Otherwise it silently falls back to German.

Reference call sites:
- `backend/app/api/graph.py:365` / `:379`
- `backend/app/api/report.py:80` / `:83`
- `backend/app/api/simulation.py:501` / `:505`
- `backend/app/services/oasis_profile_generator.py:833` / `:877`

## 6. LLM language steering

UI labels are translated by lookup, but **LLM-generated content** (the report,
agent bios/personas, section outlines) must be *produced* in the right language.
This is done by injecting the registry's `llmInstruction` into the prompts.

`get_language_instruction()` returns e.g. `"Bitte antworten Sie auf Deutsch."`
and is woven into the system/user prompts:

- **`report_agent.py`** — prompts contain a `{language_instruction}` placeholder
  filled at runtime (`:1176`, `:1272`, `:1820`). Example prompt fragment:
  > `IMPORTANT (Language): {language_instruction} The entire report outline
  > (title, summary, section titles and descriptions) MUST be written in that
  > language. The ontology entity/edge type names remain unchanged.`
- **`oasis_profile_generator.py`** — agent profile prompts inject it directly
  (`:623`, `:672`, `:722`), e.g. *"… Write the free-text fields (bio, persona)
  in that language."*

Important nuance captured in the prompts: **structural identifiers stay stable.**
Ontology entity/edge **type names are not translated** — only free-text output
(titles, summaries, bios, prose) follows the language instruction.

> Historical note: an earlier fix (`313fe64`) removed a hardcoded Chinese
> language fallback from the report agent. The `llmInstruction` mechanism is the
> clean replacement — language is now data, not hardcoded prompt text.

## 7. End-to-end request flow

```
User picks "English" in LanguageSwitcher
        │
        ├─ locale.value = 'en'  → vue-i18n re-renders all UI strings
        ├─ localStorage['locale'] = 'en'  (persists across reloads)
        │
        └─ Every axios request adds header:  Accept-Language: en
                    │
            (frontend/src/api/index.js request interceptor)
                    │
                    ▼
        Flask request → get_locale() reads 'en'
                    │
        ├─ t('api.someMessage')          → English API response
        └─ background task: locale = get_locale()  → set_locale('en') in thread
                    │
                    └─ get_language_instruction() → "Please respond in English."
                            │
                            └─ injected into LLM prompts → English report
```

The axios interceptor (`frontend/src/api/index.js`) is what bridges the two
layers:

```js
service.interceptors.request.use(config => {
  config.headers['Accept-Language'] = i18n.global.locale.value
  return config
})
```

## 8. How to add a new language

1. **Add a registry entry** in `locales/languages.json`:
   ```json
   "fr": {
     "label": "Français",
     "llmInstruction": "Veuillez répondre en français."
   }
   ```
2. **Create `locales/fr.json`** — copy `en.json` and translate every value.
   Keep the key tree **identical** to `en.json` (it's the fallback source).
3. That's it. Restart frontend (Vite picks up the new file via glob) and backend
   (loads it at import). The language appears in the switcher automatically and
   `Accept-Language: fr` resolves on the backend.

**Validation tip:** to catch drift between locale files, diff their key trees
(values will differ, structure must not):
```bash
python -c "import json;a=json.load(open('locales/en.json'));b=json.load(open('locales/fr.json'));\
def keys(d,p=''):
 for k,v in d.items():
  yield p+k
  if isinstance(v,dict): yield from keys(v,p+k+'.')
print(set(keys(a))^set(keys(b)) or 'OK')"
```

## 9. Key files reference

| File | Role |
|------|------|
| `locales/languages.json` | Language registry (label + LLM instruction) — single source |
| `locales/de.json`, `locales/en.json` | Shared translation strings (frontend + backend) |
| `frontend/src/i18n/index.js` | vue-i18n setup, glob-import, `availableLocales` export |
| `frontend/src/main.js` | Registers the i18n plugin |
| `frontend/vite.config.js` | `@` → `./src` path alias (required) |
| `frontend/src/components/LanguageSwitcher.vue` | Dropdown UI, persists choice |
| `frontend/src/api/index.js` | Adds `Accept-Language` header to every request |
| `backend/app/utils/locale.py` | Backend resolver: `t`, `get_locale`, `set_locale`, `get_language_instruction` |
| `backend/app/utils/__init__.py` | Re-exports the locale helpers |
| `backend/app/api/{graph,report,simulation}.py` | Use `t()`; capture+propagate locale into threads |
| `backend/app/services/report_agent.py` | Injects `{language_instruction}` into report prompts |
| `backend/app/services/oasis_profile_generator.py` | Injects language instruction into profile prompts |
